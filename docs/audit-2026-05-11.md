# StupidCMS — Audit & Improvement Report

**Date:** 2026-05-11
**Scope:** server/server.js, admin/app/*, configuration, multi-tenant model
**Author:** post-walkthrough notes, no code changes performed

---

## 1. The Big Picture (what's actually solid)

This is **further along than "abandoned project"** suggests:

- **460-line Express server** that handles auth, multi-tenant CRUD, Cloudinary uploads, public + admin routes, and a self-documenting `/api/meta` endpoint
- **Schema-as-code** registry (`window.COLLECTION_SCHEMAS`) — 14 collections defined
- **12 modular React components** under `admin/app/components/` (well-split, none monolithic)
- **No-build SPA** (CDN React + Babel) — admin runs without a bundler
- **Multi-tenant by folder** — physical isolation, simpler than what any commercial CMS offers at this size
- **JWT auth** with site_id binding (a token for `bobby` cannot read `rami`'s data)
- **Path traversal protection** (`site.replace(/[^a-zA-Z0-9_-]/g, '')` before joining paths)
- **Cloudinary integration** with optimized URL generation (auto-format, quality:auto)
- **Backend/frontend build coupling check** — admin warns if it's pointing at a stale server

This is roughly an MVP **already shipped**. The reframing is "polish + close gaps", not "build from scratch".

## 2. Critical Issues (block production-grade use)

### 2.1 Users hardcoded in server.js
```js
const USERS = {
  'bobby': { site_id: 'site_bobby', password: 'password' },
};
```
- **Plain text password** in source control
- **Adding a new client = git commit + redeploy** (cannot onboard via UI)
- **Single user per site** — no team support
- **Same password for everyone if you forget to change it**

**Fix:** move to `server/data/_users.json` (outside any site folder). Add bcrypt hashing. Add a `npm run user:add` CLI script to register new users without touching code.

### 2.2 JWT_SECRET fallback is dangerous
```js
const JWT_SECRET = process.env.JWT_SECRET || 'super-secret-key-change-in-production';
```
If env var isn't set on Railway, this fallback is used silently. Any attacker who knows this is the StupidCMS default can forge tokens.

**Fix:** crash on startup if `JWT_SECRET` is missing in production (`if (process.env.NODE_ENV === 'production' && !process.env.JWT_SECRET) process.exit(1)`).

### 2.3 Railway = non-persistent filesystem
Every Railway deploy wipes `server/data/`. Editor changes between deploys are lost.

**Fix options (pick one):**
1. **Auto-commit to git on every save** — server has GitHub token, runs `git add + commit + push` after each `writeJSON`. Triggers a Railway redeploy with fresh data baked in. Auto-backup + persistence in one move. Best fit for the "git is the DB" philosophy.
2. **Railway Volume** — attach persistent disk (~$1-5/month). Easy, no code change.
3. **External storage** — S3 for the JSON files. Bigger refactor.

### 2.4 No rate limiting on /api/login
Plain `POST /api/login` with no throttling. Brute-force friendly.

**Fix:** add `express-rate-limit` — 5 attempts per IP per 15 min on `/api/login`. 10 lines.

### 2.5 CORS wide open
```js
app.use(cors());  // allows any origin
```
Any website can call your CMS API. Public routes are fine, but admin routes also accept this.

**Fix:** restrict to known domains. List allowed origins in env (`ALLOWED_ORIGINS=https://kalayoga.com,https://bobby-portfolio.com`). The `/api/public/*` routes can stay open; admin routes (which use JWT) should be CORS-locked too.

## 3. Notable Bugs

### 3.1 Nav counts don't update reactively
Found live during walkthrough. `Layout.js:7` reads `localStorage.nav_counts` once at render. When `ListPage` updates a count, Layout doesn't re-render. So tab badges stay stale until navigation.

**Fix (5 min):** dispatch a `nav_counts_updated` window event from `updateCollectionCount` / `incrementCollectionCount` in helpers.js. Layout listens and re-renders. Already proposed in chat.

### 3.2 Counts only populate on visit
A collection that's never been opened shows `(0)` even if it has items. Only the **last visited** collection gets counted.

**Fix:** add `GET /api/:site/counts` endpoint that returns `{ posts: 12, faq: 4, ... }` in one call. Layout fetches on mount.

### 3.3 Build version brittle coupling
```js
const EXPECTED_BACKEND_BUILD = 'stupidcms-admin-2026-03-24';
```
A single date string. Any backend deploy that bumps the version → admin refuses to connect anywhere.

**Fix:** use semver compatibility (`backend ^X.Y.Z` matches admin's expected major). Or skip the check entirely — it's friction without much safety value.

### 3.4 Image upload size limit hardcoded
`MAX_IMAGE_UPLOAD_BYTES = 2 * 1024 * 1024` (2MB). Hero images today are often 3-5MB before Cloudinary optimizes.

**Fix:** raise to 8MB (Cloudinary handles compression on its end anyway). Make it env-driven for per-deploy override.

### 3.5 Legacy collection paths still in code
`LEGACY_COLLECTION_PATHS` (server.js:92) — technical debt from a pre-subfolder reorg. If no production data depends on these legacy paths anymore, delete them.

**Fix:** verify no live data uses legacy paths, then remove the fallback block.

| Gap | Why it matters | Effort |

| No search within collection | Hard to find a post in 50+ items | Small (filter input + client-side search) |

## 5. Improvement Ideas (worth considering)

### 5.1 Auto-commit to git on save (Highest leverage)
Replaces the Railway persistence problem AND gives you free version history, audit log, and backup. Each `writeJSON` triggers:
```
git add data/site_X/...
git commit -m "[stupidcms] bobby updated post: title"
git push origin main
```
Railway sees the push → redeploys → updated data lives in the next image.

**Bonus:** `git log --follow data/site_bobby/blog/posts.json` becomes the audit trail. Free, forever.

### 5.2 Site templates
Add a "blank" template folder under `data/_templates/`. New client onboarding = `cp -r data/_templates/blank data/site_NEWCLIENT`. Even better: an `npm run site:create -- --name kalayoga` script that does it.

### 5.3 Public RSS feed per site
`GET /api/public/:site/posts.rss` — generate Atom/RSS XML from posts.json. 30 lines. Big win for SEO and feed readers.

### 5.4 Markdown export per post
For migration / content portability. `GET /api/public/:site/posts/:slug.md` returns the post as a clean markdown file with frontmatter. Useful when a client leaves OR wants to migrate.

### 5.5 Auto-resize for image variants
After Cloudinary upload, store `{ url, url_thumb, url_medium, url_large }`. Cloudinary supports this with URL transformations — just generate the variant URLs in the upload response. Frontend gets the right size for context.

### 5.6 Per-collection access control
Schema flag: `clientCanEdit: false`. Useful for `legal` collection — you control it, client can't accidentally break their privacy policy.

### 5.7 Schema migrations
When you add/remove a field from a schema, existing JSON records are inconsistent. Add `schemaVersion` to each record + a migration runner: `npm run migrate -- posts` walks every record, applies transforms.

### 5.8 First-run onboarding
A new install with empty `data/` shows a setup wizard: create admin user, create first site, pick which collections to enable. Beats reading docs.

### 5.9 Cloudflare in front of Railway (defer indefinitely)

> Skip for now. Reasons:
> - Portfolio sites don't generate enough traffic to justify a CDN
> - Adds friction to development (cache headers confuse "why isn't my edit showing")
> - Railway handles this load class without help
>
> **Reconsider only if:** one client suddenly explodes in traffic (5,000+ daily visitors), studio grows past 30 client sites, or DDoS/scraping concerns become real.
>
> When that day comes — Cloudflare free tier in front of Railway with `Cache-Control` headers absorbs ~99% of read traffic at the edge. ~30 min setup, $0.

**Steps:**

1. **Sign up at cloudflare.com** (free plan)
2. **Add the CMS domain** (e.g., `cms.todayweb.com`) — Cloudflare gives you 2 nameservers
3. **Update domain registrar nameservers** — point them at Cloudflare's
   - *(Bonus: Cloudflare also offers domain registration at-cost. You can consolidate the studio's domains there over time. No markup, free WHOIS privacy.)*
4. **Create CNAME record**: `cms.todayweb.com` → `stupidcms-production.up.railway.app` (proxied / orange cloud ON)
5. **In Cloudflare dashboard → Caching → Configuration**: enable "Cache Everything" rule for `/api/public/*`
6. **Update server** to send proper Cache-Control headers on public routes:
   ```js
   app.get('/api/public/:site/:collection', (req, res) => {
     // ...
     res.set('Cache-Control', 'public, max-age=300, s-maxage=300');
     res.json(data);
   });
   ```

**What gets cached:** GET requests to `/api/public/*` (public reads).
**What never gets cached:** POST/PUT/DELETE, anything with `Authorization` header, `/api/login`.

**Optional purge on save:** when a record is updated in admin, server calls Cloudflare API to invalidate the relevant cache key. Otherwise edits propagate within the cache window (5 min default).

**Concrete impact:** site with 1,000 daily visitors → before: 1,000 Railway hits. After: ~24 Railway hits/day (one cache miss per region per cache window). Order-of-magnitude saving on Railway compute + bandwidth.

## 6. Security Quick-Wins (low effort, high value)

1. **bcrypt password hashing** (5 lines, 1 npm install)
2. **express-rate-limit on /api/login** (10 lines)
3. **Force JWT_SECRET to be set in prod** (3 lines)
4. **CORS allowlist for admin routes** (10 lines)
5. **HTTPS-only cookies if you ever move from localStorage to cookies**
6. **Helmet middleware** for basic security headers (1 line: `app.use(helmet())`)

Total: half a day to close all of them. None of them require architecture changes.

## 7. What NOT To Do (resist the urge)

These would push StupidCMS away from its sweet spot:

- ❌ **Add a real DB** (Postgres/Mongo) — kills the "git is the DB" simplicity. The whole point is JSON files.
- ❌ **Add a build step** (TypeScript, bundler) — kills the "no build" simplicity. CDN React + Babel is the value prop.
- ❌ **Add multi-user per tenant + roles + permissions matrix** — kills the "1 editor per site" simplicity. If you need this for one client, fork the project for them.
- ❌ **Schema-from-UI** — kills the "code is source of truth" model. Schema in code = schema in version control = aligned with how you build sites.
- ❌ **Embed a frontend framework like Next.js** — admin is intentionally separate from the public sites.

## 9. The Honest Bottom Line

StupidCMS already does **the one thing nothing on the market does cleanly**: per-tenant folder isolation with the simplicity of "edit JSON, commit to git". Every alternative we evaluated today (Decap, Tina, Sanity, Payload) was either logical multi-tenancy or expensive single-tenant. None of them beat your model for **5 portfolio sites with small blogs**.

The gaps in this audit are **polish issues**, not architectural mistakes. Architecture is right.
