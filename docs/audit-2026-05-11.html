1. The Big Picture (what's actually solid)

This is further along than "abandoned project" suggests:

  • 460-line Express server — auth, multi-tenant CRUD, Cloudinary uploads, public + admin routes, self-documenting /api/meta
  • Schema-as-code registry — 14 collections defined
  • 12 modular React components (well-split, none monolithic)
  • No-build SPA (CDN React + Babel) — admin runs without bundler
  • Multi-tenant by folder — physical isolation, simpler than any commercial CMS at this scale
  • JWT auth with site_id binding (a token for bobby cannot read rami's data)
  • Path traversal protection via regex sanitization
  • Cloudinary integration with auto-format + auto-quality URLs
  • Backend/frontend version coupling check
This is roughly an MVP already shipped. The reframing is "polish + close gaps", not "build from scratch".

2. Critical Issues (block production-grade use)

2.1 Users hardcoded in server.js

const USERS = {
  'bobby': { site_id: 'site_bobby', password: 'password' },
};
  • × Plain text password in source control
  • × Adding a new client = git commit + redeploy (cannot onboard via UI)
  • × Single user per site — no team support

Fix: move to server/data/_users.json. Add bcrypt hashing. Add npm run user:add CLI.

2.2 JWT_SECRET fallback is dangerous

const JWT_SECRET = process.env.JWT_SECRET || 'super-secret-key-change-in-production';

If env var isn't set on Railway, this fallback is used silently. Anyone who knows StupidCMS's default can forge tokens.

Fix: crash on startup if missing in prod.

2.3 Railway = non-persistent filesystem

Every Railway deploy wipes server/data/. Editor changes between deploys are lost.

Fix options (pick one):

  1. Auto-commit to git on every save — server has GitHub token, runs git add + commit + push after each writeJSON. Auto-backup + persistence. Best fit for the "git is the DB" philosophy.
  2. Railway Volume — persistent disk (~$1-5/mo). No code change.
  3. External storage — S3 for the JSON files. Bigger refactor.

2.4 No rate limiting on /api/login

Plain login route, no throttling. Brute-force friendly.

Fix: express-rate-limit — 5 attempts/IP/15min. 10 lines.

2.5 CORS wide open

app.use(cors());  // allows any origin

Fix: allowlist via env var (ALLOWED_ORIGINS=...). Public routes can stay open; admin routes should lock down.

3. Notable Bugs

3.1 Nav counts don't update reactively

Found live during walkthrough. Layout.js:7 reads localStorage.nav_counts once at render. When ListPage updates a count, Layout doesn't re-render.

Fix (5 min): dispatch nav_counts_updated event from helpers, Layout listens.

3.2 Counts only populate on visit

A collection that's never been opened shows (0) even if it has items.

Fix: add GET /api/:site/counts endpoint, fetch all on mount.

3.3 Build version brittle coupling

const EXPECTED_BACKEND_BUILD = 'stupidcms-admin-2026-03-24';

Single date string. Any backend deploy → admin refuses to connect.

Fix: use semver compatibility, or skip the check entirely.

3.4 Image upload size limit hardcoded

2MB. Hero images today are often 3-5MB pre-Cloudinary.

Fix: raise to 8MB. Make env-driven.

3.5 Legacy collection paths still in code

Technical debt from pre-subfolder reorg. If no live data needs them — delete.

5. Improvement Ideas

5.1 Auto-commit to git on save (highest leverage)

Replaces Railway persistence problem AND gives you free version history, audit log, and backup. Each writeJSON triggers a git commit + push. Railway picks up the push and redeploys.

Bonus: git log --follow data/site_bobby/blog/posts.json = audit trail.

5.2 Site templates

New client = cp -r data/_templates/blank data/site_NEW. Better: npm run site:create -- --name kalayoga.

5.3 Public RSS feed per site

GET /api/public/:site/posts.rss. ~30 lines. SEO + feed readers.

5.4 Markdown export per post

For migration / portability. Returns post as clean markdown with frontmatter.

5.5 Auto-resize image variants

After Cloudinary upload, store { url, url_thumb, url_medium, url_large } using URL transformations.

5.6 Per-collection access control

Schema flag clientCanEdit: false — useful for legal/privacy collections that you control.

5.7 Schema migrations

Add schemaVersion to each record + migration runner: npm run migrate -- posts.

5.8 First-run onboarding

Empty install shows wizard: create admin user, create first site, pick collections.

5.9 Cloudflare in front of Railway (defer indefinitely)

Skip for now. Reasons:

  • Portfolio sites don't generate enough traffic to justify a CDN
  • Cache headers add friction in development ("why isn't my edit showing?")
  • Railway handles this load class without help

Reconsider only if: one client suddenly explodes in traffic (5,000+ daily visitors), studio grows past 30 client sites, or DDoS/scraping concerns become real.

When that day comes — Cloudflare free tier in front of Railway with Cache-Control headers absorbs ~99% of read traffic at the edge. ~30 min setup, $0.

Steps:

  1. Sign up at cloudflare.com (free plan)
  2. Add the CMS domain (e.g., cms.todayweb.com) — Cloudflare gives 2 nameservers
  3. Update domain registrar nameservers — point at Cloudflare's
    • Bonus: Cloudflare offers domain registration at-cost. The studio's domains can consolidate there over time — no markup, free WHOIS privacy.
  4. Create CNAME: cms.todayweb.com → stupidcms-production.up.railway.app (proxied / orange cloud ON)
  5. Cloudflare dashboard → Caching → Configuration: enable "Cache Everything" rule for /api/public/*
  6. Update server to send Cache-Control headers on public routes: 
    app.get('/api/public/:site/:collection', (req, res) => {
  // ...
  res.set('Cache-Control', 'public, max-age=300, s-maxage=300');
  res.json(data);
});

What gets cached: GET to /api/public/*.
What never gets cached: POST/PUT/DELETE, requests with Authorization header, /api/login.

Optional purge on save: after admin edit, server calls Cloudflare API to invalidate the cache key. Otherwise edits propagate within cache window (5 min default).

Concrete impact: site with 1,000 daily visitors → before: 1,000 Railway hits. After: ~24 Railway hits/day. Order-of-magnitude saving.

6. Security Quick-Wins (low effort, high value)

  1. bcrypt password hashing — 5 lines, 1 npm install
  2. express-rate-limit on /api/login — 10 lines
  3. Force JWT_SECRET in prod — 3 lines
  4. CORS allowlist for admin routes — 10 lines
  5. Helmet middleware for security headers — 1 line

Total: half a day to close all of them.

7. What NOT To Do (resist the urge)

These would push StupidCMS away from its sweet spot:

  • × Add a real DB (Postgres/Mongo) — kills "git is the DB"
  • × Add a build step (TypeScript, bundler) — kills "no build" simplicity
  • × Multi-user + roles + permissions matrix — kills "1 editor per site" simplicity
  • × Schema-from-UI — kills "code is source of truth"
  • × Embed Next.js — admin is intentionally separate from public sites

8. Suggested Priority Order

If you give this 1 week of focused work:

  1. Day 1 — Critical security fixes (2.1, 2.2, 2.4, 2.5) + nav counts bug (3.1)
  2. Day 2 — Auto-commit-to-git (5.1) — solves Railway persistence permanently
  3. Day 3 — Site templates + onboarding script (5.2)
  4. Day 4-5 — UX gaps that hurt daily use (search, drafts, gallery view)
  5. Day 6-7 — Polish, docs, fresh deploy to Railway

9. The Honest Bottom Line

StupidCMS already does the one thing nothing on the market does cleanly: per-tenant folder isolation with the simplicity of "edit JSON, commit to git".

Every alternative we evaluated (Decap, Tina, Sanity, Payload) was either logical multi-tenancy or expensive single-tenant. None beat this model for 5 portfolio sites with small blogs.

The gaps in this audit are polish issues, not architectural mistakes. Architecture is right.
